# 龙道 AI 服务器上线前准备审计

> 审计日期：2026-07-15  
> 当前建议：先以“关闭公开注册、人工开户和充值”的封闭内测方式上线，不建议当前版本直接公开注册并在线收费。

## 1. 结论

当前项目已经具备可运行的 API 中转、用户登录、API Key、订阅/余额基础计费、后台管理和龙道 AI 品牌前端，适合在完成本文件的 P0 项后部署封闭内测环境。

当前还不具备公开商业收费所需的完整资金安全闭环。充值、支付订单、资金流水和风险事件页面目前是诚实的建设中页面，没有对应的真实支付后端、资金账本和风险事件接口。

上线方式分为两种：

1. **封闭内测上线：可以推进。** 保持公开注册关闭，由管理员创建用户、人工增加余额或发放兑换码，不开放在线支付。
2. **公开收费上线：暂不建议。** 必须先补齐支付订单、回调验签与幂等、资金账本、余额并发一致性、对账和基础风控。

## 2. 已具备并已验证的能力

- 本地 PostgreSQL、Redis、Sub2API 容器运行正常并通过健康检查。
- 后端 `go test ./...` 全量通过。
- 前端 ESLint、TypeScript 类型检查、320 个测试和生产构建通过。
- 已具备 JWT、TOTP、CORS、CSP、安全响应头、API Key IP 限制、请求体限制、限流和日志能力。
- API 使用结算已具备请求 ID 去重和事务化扣费入口：`backend/internal/repository/usage_billing_repo.go`。
- 龙道 AI 第一阶段品牌、首页、认证页、控制台、API Key 指南、套餐/充值/订单入口和管理后台入口已经完成。
- Sora 历史数据库结构缺失已通过 `backend/migrations/136_restore_sora_schema_after_legacy_drop.sql` 修复。

## 3. P0：任何服务器上线前必须完成

### 3.1 固定自有发布版本

当前 `deploy/docker-compose.yml` 使用：

```yaml
image: weishaw/sub2api:latest
```

直接在服务器运行该 Compose 会拉取上游公开镜像，**不会包含本地龙道 AI 前端、后端修改和迁移 136**。

上线前必须二选一：

- 从当前仓库构建并推送自有镜像，例如 `registry.example.com/longdao/sub2api:2026.07.15-1`；或
- 在服务器使用固定 Git 提交从源码构建镜像。

禁止生产环境长期使用未固定内容的 `latest` 标签。发布前需提交当前工作区改动，并创建可回滚的 Git tag。

### 3.2 补齐数据库升级迁移流程

首次安装时，`AUTO_SETUP` 会执行迁移；但生成 `/app/data/.installed` 后，普通重启不会再次执行 `repository.ApplyMigrations`。这会导致后续版本新增迁移时被漏掉。

上线前至少提供一种可靠方式：

- 独立的 `migrate` 命令；
- 独立迁移容器/发布脚本；或
- 启动前安全执行带校验和的幂等迁移。

发布顺序应固定为：

```text
数据库备份 → 执行迁移 → 启动新应用 → 健康检查和冒烟测试 → 放量
```

迁移失败时必须停止发布，不能带着半完成结构启动应用。

### 3.3 生产部署文件和网络边界

必须新增或整理龙道 AI 专用生产部署文件，至少满足：

- 应用只监听宿主机 `127.0.0.1:8080`，不直接暴露公网。
- 由 Nginx 或 Caddy 提供域名、HTTPS 和反向代理。
- PostgreSQL、Redis 不映射公网端口。
- 配置反向代理真实 IP 和 `trusted_proxies`，避免限流/IP 风控被伪造。
- 防火墙只开放 SSH、HTTP、HTTPS；SSH 建议限制来源并使用密钥登录。
- 为容器增加 CPU、内存和 Docker 日志大小限制。
- 健康检查失败后能够自动重启，并保留最近发布版本用于回滚。

### 3.4 固定生产密钥

生产环境必须生成并安全保存：

- `POSTGRES_PASSWORD`
- `REDIS_PASSWORD`
- `ADMIN_PASSWORD`
- `JWT_SECRET`
- `TOTP_ENCRYPTION_KEY`

同时完成：

- 管理员邮箱改为正式邮箱。
- 管理员密码使用独立强密码，不复用本地密码。
- 管理员启用 TOTP。
- `SERVER_MODE=release`、`RUN_MODE=standard`。
- 配置真实域名 CORS 白名单、CSP 和可信代理。
- 密钥只放服务器密钥文件或 Secret 管理系统，不进入 Git、镜像和公开日志。

### 3.5 备份、恢复和回滚

至少建立：

- PostgreSQL 每日自动 `pg_dump`，并保留多个历史版本。
- `/app/data` 配置和必要文件备份。
- 备份保存到另一块磁盘或对象存储，不能只留在同一台服务器。
- 上线前完成一次恢复演练，确认备份不是“只能生成、不能恢复”。
- 发布前数据库快照、旧镜像 tag 和旧 Compose 文件保留。

### 3.6 发布前冒烟验证

每次部署至少验证：

- `/health` 返回 200。
- 管理员登录和 TOTP 正常。
- 创建测试用户和 API Key 正常。
- 使用测试 Key 完成一次非流式和一次流式 API 调用。
- 用量日志、余额/套餐消耗与预期一致。
- 注册关闭时不能通过公开接口绕过注册开关。
- 重启应用后 JWT、TOTP、数据库和 Redis 数据保持正常。

## 4. P0：公开注册并在线收费前必须开发

### 4.1 支付订单闭环

需要真实实现：

- `payment_orders` 订单模型和状态机。
- 创建订单、拉起支付、查询订单接口。
- 支付回调验签、渠道事件 ID 去重和幂等处理。
- 金额、币种、商户订单号和渠道订单号校验。
- 超时关闭、重复通知、延迟通知和异常补偿。
- 退款接口、退款状态和失败重试。

### 4.2 不可篡改资金账本

必须增加独立资金流水，至少记录：

- 充值、赠送、消费、退款、人工调整、套餐购买。
- 业务单号、幂等键、变动前余额、变动金额、变动后余额。
- 操作人、来源 IP、备注和时间。

余额变动和流水写入必须在同一数据库事务中完成。当前管理员余额调整先更新用户余额，再尽力写兑换码记录；流水写入失败只记录日志，不满足正式财务审计要求。

### 4.3 余额并发和透支规则

当前请求开始前只检查 `balance > 0`，最终结算允许余额变为负数；这是源码中明确的透支策略。它能保证当前请求完成，但不适合作为公开预付费平台的最终规则。

公开收费前需确认并实现：

- 是否允许小额透支。
- 预估费用、额度预留和最终结算规则。
- 多并发请求下的原子扣费或余额预占。
- 余额不足、流式中断、上游失败和用量缺失时的补偿策略。
- 并发扣费和重复结算自动化测试。

### 4.4 支付对账和异常处理

需要每日校验：

```text
用户余额
= 初始余额
+ 充值和赠送
- API 消费和套餐购买
+ 退款和人工返还
```

并核对支付渠道订单、平台订单、资金流水、API 使用日志和余额变动。发现差异后要有告警和人工修复流程。

### 4.5 注册与基础风控

若开放公开注册，至少需要：

- 邮箱发送服务和邮箱验证。
- Turnstile/验证码和注册频率限制。
- 同 IP、同设备、同邮箱域名的异常注册限制。
- 登录失败、撞库、API Key 爆破和异常调用频率告警。
- 支付回调重放、异常充值、短时高额消费和多账号关联检测。
- 管理员敏感操作审计。

## 5. 封闭内测可以延期的功能

在保持公开注册关闭、人工开户和人工充值的前提下，以下功能可在服务器上线后迭代：

- 在线支付宝、微信、Stripe 等支付通道。
- 自动退款和完整财务对账后台。
- 完整风险规则引擎和风险事件后台接口。
- 代理商、团队、子账号和复杂角色权限。
- 发票、优惠券、多币种和多支付通道。
- 自动扩缩容和多机高可用。
- 高级运营统计和复杂报表。

占位页面必须继续明确显示“建设中”，不能让用户误以为付款后会自动到账。

## 6. 推荐的封闭内测配置

- `registration_enabled=false`。
- 管理员创建白名单用户。
- 管理员人工增加少量余额或发放兑换码。
- 用户数量和 API 并发先限制在小规模。
- 不公开充值入口或保持按钮不可用。
- 每个用户设置合理的 API Key 配额、模型白名单、并发数和 IP 限制。
- 每日检查负余额、异常消耗、上游错误和余额调整记录。
- 每日数据库备份，发布前额外备份。

## 7. 下一步推荐开发顺序

```text
1. 固化 Git 提交和自有版本镜像
2. 制作龙道 AI 生产 Compose + Caddy/Nginx 配置
3. 增加可靠的数据库 migrate 命令/发布脚本
4. 增加备份、恢复、回滚和服务器冒烟脚本
5. 验证并收紧余额并发、透支和人工调账一致性
6. 部署封闭内测环境
7. 开发资金账本
8. 开发支付订单、回调、退款和对账
9. 开发注册、支付和 API 基础风控
10. 完成公开商业上线审计
```
