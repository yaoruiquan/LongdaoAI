## 1. 背景

当前项目已完成龙道 AI 第一阶段前端基础改版，并具备用户登录、API Key、余额和订阅基础计费、API 中转、使用记录、管理员后台等基础能力。

项目即将从本地开发环境部署到服务器。上线前需要明确：

1. 哪些能力是任何服务器部署前必须完成的。
2. 哪些能力是封闭内测可以暂缓的。
3. 哪些能力是公开注册和在线收费前必须完成的。
4. 生产部署、数据库升级、备份恢复和版本回滚应满足什么标准。
5. 如何证明一个版本满足上线条件。

本 Spec 将服务器上线拆分为两种目标模式：

- **模式 A：封闭内测上线**
- **模式 B：公开商业上线**

模式 A 是当前推荐的首个服务器上线目标；模式 B 必须在支付、资金账本、计费一致性和风控闭环完成后才能启用。

---

## 2. 目标

### 2.1 产品目标

将当前本地运行的龙道 AI 平台稳定、可重复、可回滚地部署到服务器，并支持小规模白名单用户使用 API 中转服务。

### 2.2 工程目标

上线系统必须满足：

- 使用龙道 AI 自有版本镜像，不依赖内容不固定的上游 `latest` 镜像。
- 数据库结构可以在首次部署和后续升级时可靠迁移。
- 生产密钥固定、安全保存且不会进入 Git 或镜像。
- 应用、PostgreSQL、Redis 和反向代理网络边界清晰。
- 数据可以备份、恢复，应用可以回滚。
- 发布过程可以自动或半自动执行并留下验证证据。
- API 调用、用量记录和余额/套餐消耗可以完成基本一致性验证。

### 2.3 当前阶段成功标准

当前阶段以完成“模式 A：封闭内测上线”为成功标准。

成功后应达到：

- 生产域名可通过 HTTPS 访问。
- 管理员可以登录并创建白名单用户。
- 用户可以创建 API Key 并完成真实 API 请求。
- API 用量可以记录并进行余额或套餐结算。
- 在线支付保持关闭，用户不会误认为可以自动充值到账。
- 数据库每日备份，发布失败时可以回滚。

---

## 3. 非目标

以下内容不属于模式 A 的上线阻塞项：

- 在线支付宝、微信支付、Stripe、PayPal 等支付通道。
- 自动退款。
- 发票系统。
- 完整财务对账后台。
- 完整风险规则引擎。
- 代理商、团队、子账号和复杂角色权限。
- 多币种和多支付通道。
- 自动扩缩容、多节点高可用和跨区域容灾。
- 高级运营统计和复杂报表。

这些功能可以在模式 A 上线后继续开发，但在模式 B 公开商业上线前，支付和资金安全相关能力必须完成。

## 4. 术语与上线模式

### 4.1 封闭内测上线

封闭内测指：

- `registration_enabled=false`。
- 仅管理员创建用户。
- 仅白名单用户可以使用平台。
- 余额由管理员人工调整或通过兑换码发放。
- 不接入真实在线支付。
- 用户规模、API 并发和可用模型受到限制。

### 4.2 公开商业上线

公开商业上线指至少启用以下一项：

- 面向公众开放注册。
- 用户可以在线支付充值。
- 平台面向不特定用户公开销售 API 服务。
- 平台对余额、套餐、支付和退款承担正式财务责任。

### 4.3 发布门禁

发布门禁指版本进入目标环境前必须满足的检查条件。任何 P0 门禁失败时，发布必须停止。

## 5. 当前系统基线

### 5.1 已具备能力

- Go + Gin 后端。
- Vue 3 + Vite 前端。
- PostgreSQL 数据库。
- Redis 缓存和部分并发控制。
- 用户登录、JWT、刷新令牌、TOTP 和 OAuth 基础能力。
- API Key 创建、配额、有效期和 IP 限制。
- 用户余额和订阅基础结构。
- 用户组、倍率和模型权限配置。
- OpenAI、Claude、Gemini 等兼容 API 中转。
- API 使用日志和请求结算入口。
- 管理员用户、分组、账号和订阅管理。
- CORS、CSP、安全响应头、请求体限制、限流和日志能力。
- 龙道 AI 品牌首页、登录注册、用户控制台、套餐、充值、订单和管理入口。

### 5.2 当前验证基线

截至 2026-07-15：

- 后端 `go test ./...` 通过。
- 前端 ESLint 通过。
- 前端 TypeScript 类型检查通过。
- 前端 320 个测试通过。
- 前端生产构建通过。
- 生产 Docker Compose 配置语法验证通过。
- 本地 PostgreSQL、Redis、Sub2API 容器状态为 healthy。
- `/health` 返回 HTTP 200。

### 5.3 当前缺口

- 生产 Compose 仍默认使用 `weishaw/sub2api:latest`。
- 当前龙道 AI 修改尚未形成固定、可回滚的发布版本。
- 后续版本数据库迁移缺少独立、可靠的升级入口。
- 在线支付、支付订单、回调和退款后端尚未实现。
- 独立资金账本尚未实现。
- 管理员人工调整余额和审计记录不是同一事务。
- 当前余额结算策略允许最终余额变为负数。
- 风险事件页面尚无真实后端数据闭环。
- 生产 HTTPS、备份恢复、监控告警和安全加固尚未完成。

## 6. 模式 A：封闭内测上线需求

以下需求全部属于模式 A 的 P0 上线要求。

### REL-001 自有版本镜像

**需求**

系统必须使用包含龙道 AI 本地修改的自有镜像或固定源码版本部署。

**实现要求**

- 镜像名称应使用自有仓库，例如：

```text
registry.example.com/longdao/sub2api:<version>
```

- 镜像必须使用不可变版本号，不得仅使用 `latest`。
- 镜像必须包含当前前端生产构建、后端代码和全部数据库迁移。
- 镜像构建必须基于已提交的 Git commit。
- 发布版本必须创建 Git tag，并记录 commit SHA。

**验收标准**

- 服务器运行镜像不是 `weishaw/sub2api:latest`。
- 镜像版本可以映射到唯一 Git commit。
- 新服务器使用该版本部署后可以看到龙道 AI 品牌页面。
- 镜像内包含 `136_restore_sora_schema_after_legacy_drop.sql`。
- 可以切换到上一个镜像版本完成应用回滚。

### REL-002 工作区提交与版本冻结

**需求**

部署前必须将本地改动整理为可审查、可重复构建的版本。

**实现要求**

- 不丢弃当前已有前端、后端、迁移和文档修改。
- 检查未跟踪文件，确认不存在密码、Token、私钥和生产 `.env`。
- 按功能整理提交。
- 创建首个龙道 AI 服务器部署 tag。

**验收标准**

- 生产构建基于干净或已知状态的 Git 工作区。
- 服务器部署记录中包含 tag、commit 和镜像 digest。
- 相同 commit 可以重复构建出功能一致的镜像。

### MIG-001 数据库迁移命令

**需求**

系统必须支持在首次安装和后续版本升级时执行数据库迁移。

**当前问题**

首次 `AUTO_SETUP` 会执行迁移，但 `/app/data/.installed` 创建后，普通应用重启不会再次自动调用 `repository.ApplyMigrations`。

**实现方案**

至少实现以下一种：

1. 应用提供独立 `migrate` 命令。
2. 提供独立 migration 容器。
3. 提供发布脚本，在启动新应用前执行迁移。
4. 应用每次启动前安全执行带校验和的幂等迁移。

**行为要求**

- 迁移按文件名顺序执行。
- 已执行迁移必须校验 checksum。
- 新迁移必须记录到 `schema_migrations`。
- 迁移失败必须返回非零退出码。
- 迁移失败后不得启动新应用版本。
- 不允许直接修改已经在生产执行过的迁移文件。

**验收标准**

- 全新数据库可以执行全部迁移。
- 已有数据库可以只执行新增迁移。
- 重复执行迁移不会重复修改数据。
- checksum 不一致时发布被阻止。
- 迁移失败后数据库不会处于半完成状态。

### MIG-002 发布前数据库备份

**需求**

任何包含数据库结构变化的发布，必须先生成可恢复备份。

**验收标准**

- 迁移前自动或人工确认 PostgreSQL 备份成功。
- 备份文件包含时间、环境和发布版本信息。
- 备份被复制到服务器之外的存储位置。
- 至少完成一次从备份恢复到测试数据库的演练。

### DEP-001 生产 Docker Compose

**需求**

项目必须提供龙道 AI 专用生产 Compose 文件。

**服务范围**

- `sub2api`
- `postgres`
- `redis`
- 可选：`caddy` 或 `nginx`
- 可选：备份任务容器

**配置要求**

- 应用使用固定自有镜像版本。
- 应用端口仅绑定 `127.0.0.1`，或仅通过反向代理容器网络访问。
- PostgreSQL 不映射公网端口。
- Redis 不映射公网端口。
- 服务配置健康检查和自动重启。
- 容器配置 CPU、内存和日志限制。
- 数据目录使用明确命名的数据卷或宿主机目录。
- 环境变量通过生产 `.env` 或 Secret 注入。

**验收标准**

- `docker compose config -q` 通过。
- 公网只能访问 80/443，不能直接访问 PostgreSQL、Redis 和应用内部端口。
- 应用、PostgreSQL、Redis 健康检查均通过。
- 服务器重启后服务可以自动恢复。

### DEP-002 HTTPS 和反向代理

**需求**

所有用户和管理员流量必须通过 HTTPS。

**实现要求**

- 使用 Caddy 或 Nginx 终止 TLS。
- HTTP 自动跳转 HTTPS。
- 配置真实域名证书。
- 正确透传 `X-Forwarded-For`、`X-Forwarded-Proto` 和 Host。
- 后端配置可信代理范围。
- 配置合理的连接、读取和流式响应超时。
- 不得缓冲需要实时返回的流式 API 响应。

**验收标准**

- 域名 HTTPS 访问正常。
- HTTP 自动跳转 HTTPS。
- 浏览器证书链正常。
- 流式 API 可以持续输出，不被代理缓存或截断。
- 后端日志中的客户端 IP 为真实来源 IP。

### SEC-001 固定生产密钥

**需求**

生产环境必须使用固定、随机生成、独立保存的密钥。

**必要变量**

```text
POSTGRES_PASSWORD
REDIS_PASSWORD
ADMIN_PASSWORD
JWT_SECRET
TOTP_ENCRYPTION_KEY
```

**配置要求**

- `JWT_SECRET` 和 `TOTP_ENCRYPTION_KEY` 不得为空。
- 管理员密码不得复用本地开发密码。
- 管理员邮箱必须改为正式邮箱。
- 管理员必须启用 TOTP。
- `SERVER_MODE=release`。
- `RUN_MODE=standard`。
- 生产密钥不得写入 Git、Dockerfile、镜像层和公开日志。

**验收标准**

- 应用重启后原登录状态按预期保持或过期，不因随机 JWT Secret 全部失效。
- 应用重启后已绑定 TOTP 仍然可用。
- Redis 未授权访问失败。
- 数据库未授权访问失败。
- Git 历史和发布镜像中不存在生产密钥。

### SEC-002 网络和主机安全

**需求**

服务器只开放业务必要端口，并限制管理入口。

**实现要求**

- 防火墙只开放 SSH、HTTP、HTTPS。
- SSH 使用密钥认证，建议关闭密码登录。
- 管理后台可增加 IP 白名单、VPN 或额外认证。
- 配置 CORS 生产域名白名单。
- 配置可信代理。
- 保持 CSP、安全响应头和请求体限制启用。

**验收标准**

- 外网扫描无法访问 PostgreSQL 和 Redis。
- 非白名单 Origin 无法跨域调用受保护接口。
- 管理接口需要管理员身份。
- 客户端无法通过伪造代理请求头绕过 IP 限制。

### OPS-001 自动备份

**需求**

生产数据必须自动备份，并具备明确保留周期。

**备份范围**

- PostgreSQL。
- `/app/data` 中必要配置和持久化文件。
- 生产部署文件，但不将明文密钥复制到不安全位置。

**最低策略**

- PostgreSQL 每日备份。
- 发布前额外备份。
- 保留多个历史版本。
- 至少一份备份保存到服务器之外。

**验收标准**

- 备份任务可以被监控。
- 备份失败会产生告警。
- 随机选择一个备份可以恢复到测试环境。
- 恢复后的用户、API Key、余额、套餐和用量数据可读取。

### OPS-002 回滚能力

**需求**

应用发布失败时必须可以快速回滚。

**实现要求**

- 保留上一版本镜像。
- 保留上一版本 Compose 和反向代理配置。
- 数据库迁移默认采用前向修复；涉及不兼容迁移时必须提前设计回滚或恢复方案。
- 回滚脚本不得删除生产数据卷。

**验收标准**

- 可以在不删除 PostgreSQL 和 Redis 数据的情况下切换回上一应用版本。
- 回滚后 `/health`、登录和 API 调用正常。
- 回滚操作和结果被记录。

### OPS-003 日志、监控和告警

**需求**

封闭内测期间必须能及时发现服务不可用和异常消耗。

**最低监控项**

- 应用存活和 `/health`。
- 容器重启次数。
- CPU、内存和磁盘使用率。
- PostgreSQL、Redis 状态。
- API 请求量、错误率和延迟。
- 上游账号错误率。
- 用户负余额。
- 异常余额调整。
- 备份任务状态。

**验收标准**

- 健康检查失败能够通知管理员。
- 磁盘空间、数据库连接和错误率达到阈值时能够告警。
- Docker 和应用日志有大小或保留周期限制。

### UAT-001 封闭内测业务规则

**需求**

模式 A 必须强制执行封闭内测规则。

**规则**

- `registration_enabled=false`。
- 用户由管理员创建。
- 余额由管理员人工调整或兑换码发放。
- 在线充值按钮保持禁用或明确提示尚未开放。
- 支付订单、资金流水、风险事件页面保持建设中状态。
- 每个用户设置 API Key 配额、模型权限、并发限制和可选 IP 限制。

**验收标准**

- 未登录访客不能完成注册。
- 关闭注册时不能通过直接调用接口绕过。
- 用户点击充值不会创建真实支付订单或错误增加余额。
- 管理员可以创建用户并人工发放测试额度。
- 白名单用户可以完成 API 调用。

### TST-001 发布冒烟测试

**需求**

每次部署后必须执行自动或半自动冒烟测试。

**测试项**

1. `/health` 返回 HTTP 200。
2. 首页、登录页和控制台静态资源正常。
3. 管理员登录正常。
4. 管理员 TOTP 正常。
5. 创建测试用户正常。
6. 创建 API Key 正常。
7. 非流式 API 请求正常。
8. 流式 API 请求正常。
9. 用量日志正常。
10. 余额或套餐消耗正常。
11. 应用重启后数据正常。
12. 注册关闭策略正常。

**验收标准**

- 所有 P0 冒烟项目通过。
- 失败项目会终止发布或阻止放量。
- 测试输出包含版本、时间和环境信息。

## 7. 模式 B：公开商业上线需求

模式 B 包含模式 A 的全部要求，并新增以下 P0 要求。

### PAY-001 支付订单模型

**需求**

系统必须建立独立支付订单模型和状态机。

**必要字段**

- 平台订单号。
- 用户 ID。
- 支付渠道。
- 金额和币种。
- 订单状态。
- 渠道订单号。
- 创建、支付、关闭和退款时间。
- 幂等键。
- 原始回调摘要或事件 ID。

**状态要求**

至少支持：

```text
pending
paid
closed
failed
refunding
refunded
```

**验收标准**

- 状态转换受约束，不能任意跳转。
- 平台订单号唯一。
- 幂等键唯一。
- 已支付订单不能被重复入账。

### PAY-002 创建支付和查询订单

**需求**

用户可以创建支付订单并查询订单状态。

**验收标准**

- 金额由服务端套餐或充值规则计算，不能信任前端提交金额。
- 重复请求不会创建多笔不可控订单。
- 用户只能查询自己的订单。
- 管理员可以查询全部订单。

### PAY-003 支付回调安全

**需求**

支付渠道回调必须安全、幂等地处理。

**实现要求**

- 验证签名。
- 验证商户号、应用 ID、金额、币种和订单号。
- 渠道事件 ID 或回调 ID 唯一。
- 重复回调返回成功，但不重复增加余额。
- 回调原文或必要摘要可审计。
- 回调处理失败支持重试和人工补偿。

**验收标准**

- 伪造签名被拒绝。
- 篡改金额被拒绝。
- 相同回调重复发送不会重复入账。
- 回调处理与余额入账在可靠事务边界内完成。

### LEDGER-001 独立资金账本

**需求**

所有余额变化必须写入独立、可追溯的资金账本。

**流水类型**

- 充值。
- 赠送。
- API 消费。
- 套餐购买。
- 退款。
- 人工增加。
- 人工扣减。
- 异常补偿。

**必要字段**

- 流水号。
- 用户 ID。
- 业务类型。
- 业务单号。
- 幂等键。
- 变动前余额。
- 变动金额。
- 变动后余额。
- 操作人。
- 来源 IP。
- 备注。
- 创建时间。

**一致性要求**

- 余额变动和资金流水必须在同一事务中提交。
- 流水记录不可直接修改或删除；更正应创建反向或补偿流水。
- 业务单号和幂等键需要唯一约束。

**验收标准**

- 每次余额变化都能找到对应流水。
- 流水前后余额连续。
- 人工调账失败时余额和流水同时回滚。
- 重放相同业务请求不会重复变动余额。

### BILL-001 余额预占与并发一致性

**需求**

公开预付费模式必须明确并实现余额不足、预占、最终结算和透支规则。

**当前行为**

当前系统请求开始前只检查 `balance > 0`，请求结束后允许余额变成负数。

**待确认产品规则**

- 是否允许透支。
- 最大允许透支额度。
- 是否对请求进行费用预估。
- 是否预占余额。
- 流式响应中断时如何结算。
- 上游失败和平台失败是否收费。
- 无法获得精确用量时如何处理。

**工程要求**

- 多并发请求不能绕过余额限制产生无上限透支。
- 预占、结算和释放必须具备幂等性。
- 最终扣费必须与用量记录关联。
- API Key 配额和用户余额更新应在可靠事务边界内执行。

**验收标准**

- 并发扣费测试通过。
- 重复结算测试通过。
- 流式断开测试通过。
- 上游失败测试通过。
- 余额不足行为与产品规则一致。

### BILL-002 计费快照

**需求**

每次 API 结算必须保存当时使用的计费规则快照。

**快照内容**

- 模型标识。
- 输入和输出 Token 数。
- 缓存 Token 数。
- 原始单价。
- 用户组倍率。
- 服务层级倍率。
- 套餐或余额计费模式。
- 最终费用。
- 规则版本。

**验收标准**

- 后续修改价格不会改变历史费用。
- 任意一笔 API 消费可以重新计算并解释最终费用。

### RECON-001 每日对账

**需求**

系统必须执行支付、资金和 API 消费对账。

**对账公式**

```text
用户当前余额
= 初始余额
+ 所有充值和赠送
- 所有 API 消费和套餐购买
+ 所有退款和人工返还
```

**核对范围**

- 支付渠道成功订单与平台支付订单。
- 平台支付订单与资金流水。
- API 使用日志与计费快照。
- 计费快照与消费流水。
- 消费流水与用户余额。

**验收标准**

- 每日对账可以输出差异清单。
- 差异产生告警。
- 每个差异可以被人工确认和补偿。

### REFUND-001 退款与异常补偿

**需求**

系统必须处理全额退款、部分退款和异常补偿。

**验收标准**

- 退款请求幂等。
- 退款状态可查询。
- 退款成功后生成对应资金流水。
- 重复退款被阻止。
- 支付渠道退款与平台余额返还可以对账。

### RISK-001 注册和登录风控

**需求**

公开注册必须具备最低限度的反滥用能力。

**最低规则**

- 邮箱验证。
- Turnstile 或同类人机验证。
- IP 注册频率限制。
- 邮箱和设备维度频率限制。
- 登录失败次数限制。
- 异常登录告警。

**验收标准**

- 高频注册被限制。
- 验证码绕过测试失败。
- 登录爆破触发限制或告警。

### RISK-002 API 和支付风控

**最低规则**

- API Key 爆破检测。
- 单用户短时高并发和高消费检测。
- 多账号共用 IP 或设备异常检测。
- 支付回调重放检测。
- 异常充值金额和频率检测。
- 负余额和余额突变告警。
- 管理员敏感操作审计。

**验收标准**

- 风险事件能够落库。
- 管理后台能够查询和处理风险事件。
- 高风险规则能够阻止操作或产生告警。

- 备份存储与生产服务器故障域分离。

## 10. 发布流程

### 10.1 标准发布流程

```text
1. 确认 Git 工作区和版本 tag
2. 构建自有镜像
3. 执行后端测试
4. 执行前端 lint、类型检查、测试和构建
5. 推送固定版本镜像
6. 备份生产数据库
7. 执行数据库迁移
8. 更新生产 Compose 镜像版本
9. 启动新版本
10. 执行健康检查
11. 执行登录和 API 冒烟测试
12. 检查日志、余额和用量
13. 小流量放量
14. 记录发布结果
```

### 10.2 发布失败处理

- 镜像启动失败：回滚上一镜像。
- 健康检查失败：停止放量并回滚。
- 数据库迁移失败：停止启动新应用，根据事务状态恢复或使用备份。
- API 计费异常：立即关闭外部流量，保留日志和数据后回滚。
- 数据损坏：停止写入并执行恢复流程。

## 15. 已知风险和待确认项

### 15.1 已知风险

- 当前生产 Compose 使用上游 `latest` 镜像，直接部署会丢失本地改造。
- 当前工作区存在大量未提交修改，尚未形成正式发布版本。
- 数据库后续升级可能漏执行新增迁移。
- 当前余额结算允许透支。
- 管理员余额调整与调整记录不是同一事务。
- 支付和风险相关页面尚无真实后端闭环。
- 尚未执行生产压力测试和数据库恢复演练。

### 15.2 待确认产品规则

在开发模式 B 前，需要明确：

- 目标用户地区。
- 结算币种。
- 首个支付渠道。
- 是否允许退款和部分退款。
- 是否允许余额透支。
- 套餐优先还是余额优先。
- 注册是否需要邮箱、邀请码或实名认证。
- 是否提供注册赠送、推广奖励或优惠码。
- 首版模型范围和模型价格。

### 15.3 验证缺口

- 本机未安装 `govulncheck`，尚未完成 Go 官方漏洞扫描。
- 当前 `pnpm audit` 因 registry audit 端点返回 HTTP 410 未完成。
- 尚未进行长时间流式调用和高并发计费压测。
- 尚未完成真实生产服务器的断网、重启、磁盘满和数据库恢复演练。
