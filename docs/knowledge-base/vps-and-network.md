# VPS 选型与网络基础

> 面向有一定开发基础、但还没有独立运维过服务器的开发者。

---

## 1. VPS 是什么

VPS（Virtual Private Server，虚拟私有服务器）是一台"独占资源"的虚拟机，跑在物理服务器上，但对外表现得像一台独立机器：你有 root 权限、固定 IP、随时在线。

和其他托管形式的区别：

| 形式 | 类比 | 适合场景 |
|------|------|----------|
| 共享主机 | 合租公寓，厨房共用 | 静态网站，流量低 |
| 云函数（Serverless） | 钟点工，按次计费 | 短任务，突发流量 |
| **VPS** | **整租一室户，24h 在线** | **长期运行的服务，如 API 中转站** |

LLM 中转站需要维持 WebSocket/SSE 长连接、响应流式请求，云函数有超时限制（通常 15-30 秒），而一个长文提示的补全可能要 60 秒以上——VPS 是更合适的选择。

## 2. 选服务器的关键维度

### 2.1 地区的影响：延迟与 IP 声誉

地区影响两件事：**从中国大陆到服务器的速度**，以及**服务器 IP 能否访问 OpenAI/Claude 等官方 API**。

- **香港**：离大陆最近，延迟低（约 5-15ms），但香港 IP 段已被 OpenAI 大量封禁，直接用香港 IP 调用官方 API 会报 `403 country not supported`。
- **日本**：略远（大陆到东京约 50-80ms），但日本 IP 声誉相对干净，大多数聚合中转站和官方 API 均可访问。
- **美国/欧洲**：IP 声誉最好，但延迟高，中间链路也更复杂。

本次选择了**日本 VPS**，就是因为香港 IP 被封，日本是最近的"干净"选项。

### 2.2 线路类型

线路影响的是"数据包从中国出境走哪条物理光缆"。

| 线路 | 说明 | 特点 |
|------|------|------|
| **CN2 GIA** | 电信自建高端线路，Global Internet Access | 延迟低、抖动小、稳定，贵 |
| **软银（SoftBank）** | 日本软银 BGP 线路 | 质量次于 CN2 GIA，但稳定，适合日本节点 |
| **普通 BGP** | 走普通互联网骨干 | 高峰期可能拥堵，便宜 |

对于 LLM 中转站这个场景：用户不在中国大陆访问（或者客户都在海外），线路类型影响不大；如果主要服务大陆用户，CN2 GIA 或软银会显著改善体验。

### 2.3 带宽 vs 流量：哪个才是瓶颈？

这是最容易混淆的两个指标：

- **带宽**（Bandwidth）：水管的粗细，单位 Mbps。5 Mbps 意味着每秒最多传输 0.625 MB 数据。
- **流量**（Traffic）：一个月里总共能流多少水，单位 GB/月。

**LLM 文字转发的实际需求估算：**

一次对话请求（输入+输出合计约 2000 token）的响应体大约 3-8 KB。即便是流式传输（SSE），单个请求的数据量也很小。

真正的挑战是**并发**：10 个用户同时在流式接收响应，每个连接持续 10 秒，同时占用的带宽才是瓶颈。

```
并发 10 个流式请求 × 每请求约 50 KB/s = 500 KB/s ≈ 4 Mbps
```

所以 5 Mbps 固定带宽理论上勉强够，但没有余量。而 100 Mbps 带宽 + 500 GB/月流量的方案：
- 带宽充足，支持 100+ 并发流式请求
- 500 GB 流量对文字 API 转发来说极度宽裕（纯文本按月消耗通常不超过 20 GB）

**本次选择日本 100 Mbps/500 GB 而非香港 5 Mbps 的技术理由**：
1. 香港 IP 被 OpenAI 封禁，根本无法用
2. 100 Mbps 带宽为并发流式请求留足余量
3. 未来扩容用户时不会立刻遇到带宽墙

## 3. 域名和 DNS 基础

### 3.1 A 记录：把域名指向 IP

DNS（Domain Name System）就是互联网的电话簿。A 记录是最基础的记录类型：

```
longdaoai.cn  A  12.34.56.78
```

意思是"当有人访问 longdaoai.cn 时，告诉他去找 IP 12.34.56.78"。

### 3.2 DNS 传播时间

改了 A 记录，为什么要等几分钟甚至更久才生效？

全球有无数 DNS 缓存服务器，它们会把查询结果缓存一段时间（TTL，Time To Live）。改完 A 记录后：
- 你本机的 DNS 缓存：可以 `ipconfig /flushdns`（Windows）或重启网络 flush 掉
- 运营商的递归 DNS：等 TTL 过期，通常 5 分钟 ~ 1 小时

把 TTL 改小（如 60 秒）可以加速传播，但大多数域名注册商默认 TTL 是 600 秒（10 分钟）。

### 3.3 为什么需要域名

直接用 IP（如 `http://12.34.56.78:8080`）能访问，但：

1. **HTTPS 证书无法签给裸 IP**：Let's Encrypt 的证书需要验证你拥有该域名（DNS 验证或 HTTP 验证），无法为 IP 颁发证书。
2. **用户体验**：`longdaoai.cn` 比 `12.34.56.78` 好记。
3. **灵活迁移**：换 VPS 时只需改 A 记录，用户的 `base_url` 不需要变。

## 4. HTTPS/TLS 是怎么工作的（直觉理解）

### 4.1 证书的作用

当你访问 `https://longdaoai.cn` 时，浏览器（或 API 客户端）会问服务器："你真的是 longdaoai.cn 吗？"服务器出示证书回答："是的，这是权威机构签发给我的证明。"

证书由 CA（Certificate Authority，证书颁发机构）签发，它们被操作系统和浏览器预置信任。没有证书或证书不匹配，客户端会报 `SSL certificate error`，连接中断。

### 4.2 Let's Encrypt：免费证书自动续期

Let's Encrypt 是非盈利的免费 CA，通过 ACME 协议自动化证书申请流程：

1. Caddy 向 Let's Encrypt 说："我要申请 longdaoai.cn 的证书"
2. Let's Encrypt 说："在 `http://longdaoai.cn/.well-known/acme-challenge/xxx` 放一个特定文件来证明你控制这个域名"
3. Caddy 自动放好文件，Let's Encrypt 验证通过，颁发证书
4. 证书有效期 90 天，Caddy 会在到期前自动续期

全程无需人工干预，这是 Caddy 相比 Nginx 最大的便利之一。

### 4.3 HTTP → HTTPS 跳转

Caddy 默认将 `http://` 请求 301/308 重定向到 `https://`，原因：
- 防止客户端误用明文传输（API Key 可能泄露）
- 搜索引擎偏好 HTTPS

## 5. 反向代理（Caddy 的角色）

### 5.1 请求路径

```
用户/客户端
    │  HTTPS:443
    ▼
  Caddy（反向代理，Docker 容器）
    │  HTTP:8080（内网）
    ▼
  sub2api（应用容器）
```

Caddy 在最前面，处理 TLS 握手、证书管理、HTTP→HTTPS 跳转，然后把请求转发给内部的 sub2api 容器。

### 5.2 为什么应用不直接暴露公网

- **安全隔离**：sub2api 容器只监听 Docker 内网的 8080 端口，外部无法直接访问，攻击面更小。
- **统一入口**：Caddy 可以同时代理多个服务，只需一个公网 IP 和 443 端口。
- **灵活配置**：限速、日志、访问控制都在 Caddy 层统一处理，应用代码无需关心。

### 5.3 `X-Forwarded-For` / `X-Real-IP`

Caddy 转发请求时，原始请求的客户端 IP 会丢失（sub2api 只能看到 Caddy 的内网 IP）。为此，Caddy 在 HTTP Header 里加上：

```
X-Forwarded-For: 203.0.113.42
X-Real-IP: 203.0.113.42
```

sub2api 读取这个 header 才能知道真实的客户端 IP，用于日志记录、限速、风控等。

### 5.4 `flush_interval -1`：流式响应必须禁用缓冲

LLM 接口使用 SSE（Server-Sent Events）逐 token 输出。如果反向代理开启了响应缓冲，Caddy 会积累完整响应再转发，用户看到的就是"等了很久，然后一下子全部出来"，流式体验完全消失。

在 Caddyfile 里：

```caddyfile
reverse_proxy sub2api:8080 {
    flush_interval -1   # -1 表示立即 flush，禁用缓冲
}
```

## 6. 防火墙基础

### 6.1 只开放必要端口

服务器上的每个开放端口都是潜在的攻击入口。最小化原则：

| 端口 | 服务 | 是否对公网开放 |
|------|------|----------------|
| 22 | SSH | 是（建议改为非标准端口或用密钥认证） |
| 80 | HTTP | 是（Caddy 用于 ACME 验证和 HTTPS 跳转） |
| 443 | HTTPS | 是（主要服务端口） |
| 5432 | PostgreSQL | **否** |
| 6379 | Redis | **否** |
| 8080 | sub2api | **否**（只在 Docker 内网） |

PostgreSQL 和 Redis 只在 Docker 网络内部互通，不映射到宿主机公网端口。即便宿主机防火墙有漏洞，数据库也不会直接暴露。

### 6.2 安全组（云控制台）vs iptables

两者都是防火墙，但层次不同：

- **安全组**（云控制台配置）：在云厂商的网络层生效，流量甚至不会到达你的服务器。配置简单，界面操作，适合大多数场景。
- **iptables**（服务器内部）：在操作系统内核层生效，更灵活但配置复杂，规则写错可能把自己的 SSH 锁掉。

建议：**优先用安全组**，在云控制台界面配置入站规则即可。如果需要更精细的控制（如按 IP 段限制 SSH），再叠加 iptables 或 `ufw`。
