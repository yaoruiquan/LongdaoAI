# Sub2API Token 中转站改造路线图

> 文档状态：持续更新  
> 当前阶段：第一阶段前端基础改版已完成，进入封闭内测上线准备与生产部署加固  
> 更新日期：2026-07-15

## 1. 项目目标

在现有 Sub2API 项目基础上，逐步改造成一个可正式运营的 AI Token/API 中转平台，支持用户注册、API Key 管理、余额或套餐计费、在线支付、风控、运营后台以及服务器部署。

总体原则：

1. 先完成本地开发和测试闭环，再部署到服务器。
2. 先固定核心业务规则，再调整数据库和接入支付。
3. 所有资金、余额、配额变动必须可追溯、可对账、可幂等重放。
4. 数据库结构通过迁移脚本演进，不直接手工修改生产数据库。
5. 每个阶段完成后必须通过测试和验收，再进入下一阶段。

## 7. 阶段五：计费规则与资金账本

### 7.1 目标

建立可解释、可重放、可对账的计费核心，再接入真实支付。

### 7.2 定价模型

建议价格规则至少支持：

- 平台
- 模型
- 输入 Token 单价
- 输出 Token 单价
- 缓存写入单价
- 缓存读取单价
- 图片、音频、视频等特殊计费单位
- 用户组倍率
- 生效时间和失效时间
- 价格版本

### 7.3 计费公式

示例：

```text
基础费用 = 输入费用 + 输出费用 + 缓存费用 + 特殊媒体费用
用户费用 = 基础费用 × 用户组倍率 × 其他业务倍率
最终费用 = max(用户费用, 最低扣费)
```

实际公式必须在阶段一确认后固化。

### 7.4 余额和账本原则

禁止仅修改用户余额而不记录流水。每次余额变化必须生成独立账本记录。

建议流水类型：

- 在线充值
- 管理员充值
- API 消费
- API 退款
- 套餐购买
- 赠送余额
- 兑换码充值
- 订单退款
- 风控冻结
- 冻结释放
- 人工调账

每条资金流水建议包含：

- 用户 ID
- 业务类型
- 业务单号
- 变动前余额
- 变动金额
- 变动后余额
- 币种
- 幂等键
- 操作人
- 备注
- 创建时间

### 7.5 结算流程

- [ ] 请求前检查余额或套餐额度
- [ ] 必要时进行额度预留
- [ ] 请求结束后计算实际费用
- [ ] 预留金额与实际金额结算
- [ ] 上游失败时释放预留额度
- [ ] 异常中断时通过补偿任务恢复
- [ ] 所有结算操作使用数据库事务
- [ ] 使用幂等键避免重复扣费

### 7.6 验证标准

- 用户余额与流水汇总一致
- 重复提交相同结算请求不会重复扣款
- 失败请求能按规则退款或不扣费
- 并发请求不能造成余额透支
- 历史价格变更不影响旧账单解释

## 8. 阶段六：支付系统

### 8.1 目标

建立支付订单、支付回调、余额入账和退款的完整闭环。

### 8.2 推荐订单状态

- `pending`：等待支付
- `processing`：支付处理中
- `paid`：支付成功
- `expired`：订单超时
- `cancelled`：订单取消
- `refunding`：退款处理中
- `refunded`：已退款
- `failed`：支付失败

### 8.3 支付流程

1. 用户创建充值或套餐订单
2. 后端锁定订单金额和币种
3. 后端向支付渠道创建支付对象
4. 前端展示支付页面或二维码
5. 支付渠道通知后端 webhook
6. 后端验签并记录原始支付事件
7. 使用渠道交易号和事件号进行幂等检查
8. 更新订单状态
9. 创建资金流水或套餐权益
10. 返回成功并等待用户端主动查询订单状态

### 8.4 必须实现的安全要求

- [ ] 支付金额只能由后端计算
- [ ] 前端不能直接决定实际到账金额
- [ ] webhook 必须验签
- [ ] webhook 必须幂等
- [ ] 支付成功以服务端回调或服务端查询为准
- [ ] 原始支付事件需保存，用于对账和排障
- [ ] 支付密钥只保存在服务器环境变量或密钥系统
- [ ] 日志中不得输出完整签名、银行卡或支付凭据
- [ ] 订单超时由后台任务处理
- [ ] 退款必须关联原始支付订单

### 8.5 管理后台

- [ ] 支付订单列表和详情
- [ ] 按用户、渠道、状态和时间筛选
- [ ] 支付事件查看
- [ ] 人工补单需二次确认和审计
- [ ] 退款操作及退款状态
- [ ] 日、周、月支付统计
- [ ] 渠道账单对账

### 8.6 验证标准

- 重复 webhook 不会重复到账
- 伪造 webhook 无法改变订单状态
- 订单金额、支付金额和到账金额可对账
- 支付成功但业务处理失败时有补偿任务
- 沙箱环境端到端支付测试通过后才允许接入生产密钥

## 9. 阶段七：风控系统

### 9.1 目标

控制注册、登录、API 调用、支付和管理员操作风险。

### 9.2 注册和登录风控

- [ ] IP 注册频率限制
- [ ] 邮箱注册频率限制
- [ ] 登录失败次数限制
- [ ] 人机验证
- [ ] 异常地区或设备登录提醒
- [ ] 高风险账号冻结

### 9.3 API 调用风控

- [ ] 用户级 QPS、RPM、TPM 限制
- [ ] API Key 级限制
- [ ] IP 级限制
- [ ] 并发请求限制
- [ ] 单次最大 Token 限制
- [ ] 单日消费限额
- [ ] 模型白名单和黑名单
- [ ] API Key IP 白名单或黑名单
- [ ] 异常错误率和请求模式检测
- [ ] 疑似密钥泄漏自动禁用

### 9.4 支付和资金风控

- [ ] 短时间大量创建订单
- [ ] 多账号共用支付特征
- [ ] 支付后快速高额消耗
- [ ] 异常退款频率
- [ ] 管理员人工调账额度限制
- [ ] 高额充值和高额消费告警

### 9.5 风险事件

风险事件建议记录：

- 风险类型
- 风险等级
- 用户、API Key、IP 和设备标识
- 关联请求或订单
- 命中的规则
- 原始证据摘要
- 自动处置结果
- 人工复核结果
- 创建和处理时间

### 9.6 验证标准

- 限流和冻结不能只依赖单机内存
- 多实例部署时风控状态保持一致
- 风险处置有明确原因和审计记录
- 敏感风控规则不向普通用户暴露

- 资源类型和资源 ID

- IP 和 User-Agent
- 创建时间

### 10.3 数据库迁移要求

- [ ] 每次结构变更提供正向迁移脚本
- [ ] 高风险迁移提供回滚或恢复方案
- [ ] 新字段先允许空值，再回填，再添加约束
- [ ] 大表索引评估锁表影响
- [ ] 金额使用精确小数或最小货币单位整数
- [ ] 业务单号、幂等键和渠道事件 ID 设置唯一约束
- [ ] 所有外键和高频查询字段评估索引
- [ ] 上线前在数据库副本或测试数据上演练迁移

### 10.4 验证标准

- 新旧数据能够正确迁移
- 迁移脚本可以在全新数据库执行
- 迁移脚本可以在已有数据的数据库执行
- 并发资金操作通过事务和约束保证一致性
- 数据库备份可成功恢复
